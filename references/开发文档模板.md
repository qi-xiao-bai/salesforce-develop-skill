# 开发文档模板

在用户明确要求维护开发文档时，按这个模板更新项目根目录下对应的 `*-开发文档.md`。

这是一份“当前代码实现说明”风格的公共模板，重点是把当前代码实际怎么实现、实际怎么运行讲清楚，不写方案汇报，不写未来规划，不绑定具体业务场景。

## 最终标准结构

```md
# XXX - 当前代码实现说明

## 功能目标

## 当前完成情况

## 文档目的

## 代码分层与职责

## 核心类与职责

## 关键方法说明

## 关键对象、字段、状态

## 完整流程链

## 方法调用链路

## 当前实现特征与限制

## 边界处理与特殊说明

## 维护与扩展指引

## 当前已知问题与优化建议

## 错误处理现状

## 待完善项

## 已知风险或注意点

## 总结
```

## 文档标题

推荐写法：

```md
# XXX - 当前代码实现说明
```

如果是局部功能，也可以写成：

```md
# XXX功能 - 当前代码实现说明
```

## 功能目标

这一节只写当前功能要解决什么问题、要达到什么结果，不展开实现细节。

推荐写法：

```md
## 功能目标

实现或维护某个核心功能，确保入口、主流程、状态流转和后置处理满足当前业务要求。
```

## 当前完成情况

这一节只写现状，不写空泛总结。

推荐写法：

```md
## 当前完成情况

- 已完成主流程开发
- 已完成核心对象落库逻辑
- 已完成主要状态流转
- 待补充异常分支处理
- 待补充测试覆盖
```

## 文档目的

这一节回答两个问题：

- 这份文档描述的是什么范围
- 阅读者看完后应该能理解什么

推荐写法：

```md
## 文档目的

这份文档只描述当前代码实际怎么实现、实际怎么运行。

目标是让阅读者看完后，能够快速理解入口、分层职责、主链路、关键状态流转、主要限制和排查重点。
```

## 代码分层与职责

先把大结构讲清楚，再往下写核心类、流程和方法。

推荐写法：

```md
## 代码分层与职责

当前实现可以先按以下几层理解：

### 1. Controller / Entry

负责接收前端、接口、批处理或其他入口请求，完成参数包装和结果返回。

### 2. Gateway / Facade

负责统一入口和请求分发，本身尽量不承载重业务逻辑。

### 3. Service

负责主业务流程编排、前置校验、主数据处理和结果汇总。

### 4. Trigger / Handler

负责记录落库后的衍生逻辑、状态补充或同步处理。

### 5. Support / Utility

负责公共查询、公共判断、公共动作、常量和共享支撑能力。
```

## 核心类与职责

这一节只写核心类，不要求把所有类都列出来。

推荐写法：

```md
## 核心类与职责

- `XXXController`
  - 负责入口请求接收和结果返回
- `XXXService`
  - 负责主业务流程编排
- `XXXTriggerHandler`
  - 负责对象落库后的补充处理
- `XXXSupport`
  - 负责公共支撑逻辑
```

## 关键方法说明

这一节只写核心方法，不需要把每一个 helper 都展开。

推荐写法：

```md
## 关键方法说明

### `XXXController.handleSubmit(request)`
- 入参：`request`
- 出参：`Result`
- 作用：入口方法，负责接收前端请求并调用主服务
- 调用时机：用户触发提交操作时

### `XXXService.submit(commands, allOrNone)`
- 入参：`commands`、`allOrNone`
- 出参：`List<Result>`
- 作用：执行业务主链路并汇总处理结果
- 调用时机：入口层完成参数包装后
```

## 关键对象、字段、状态

如果当前功能涉及对象、字段、状态或运行时表，建议单独写这一节。

推荐写法：

```md
## 关键对象、字段、状态

### 配置类对象

- `XXXConfig__c`
  - 作用：存储流程或规则配置

### 运行时对象

- `XXXInstance__c`
  - 作用：存储一次执行过程中的主实例
- `XXXTask__c`
  - 作用：存储执行过程中的待处理项

### 关键字段

- `Status__c`
  - 作用：标识当前状态
- `TargetId__c`
  - 作用：关联目标业务记录
```

## 完整流程链

这一节先写实际运行流程，不急着堆方法名。

推荐写法：

```md
## 完整流程链

1. 调用方发起请求
2. 入口层接收请求并完成参数包装
3. 主服务执行前置校验和上下文构建
4. 创建或更新核心业务数据
5. 执行后置逻辑、通知或异步处理
6. 返回结果给调用方
```

## 方法调用链路

这一节是强制项。

不要只写方法名列表。每一行都要补中文解释，写清：

- 入参
- 出参
- 作用

推荐写法：

```md
## 方法调用链路

整体链路说明：
请求先进入入口层，再进入主服务完成核心编排，随后执行落库和后置处理，最后返回结果。

请求进入系统
-> `XXXController.handleSubmit(request)`
   - 入参：`request`
   - 出参：`Result`
   - 作用：接收请求并调用主服务
-> `XXXGateway.process(command, allOrNone)`
   - 入参：`command`、`allOrNone`
   - 出参：`ProcessResult`
   - 作用：统一分发请求到后续服务
-> `XXXService.submit(commands, allOrNone)`
   - 入参：`commands`、`allOrNone`
   - 出参：`List<Result>`
   - 作用：执行业务主链路并汇总处理结果
-> `XXXService.buildContext(command)`
   - 入参：`command`
   - 出参：`Context`
   - 作用：完成前置校验和上下文构建
-> `XXXService.persistCoreData(context)`
   - 入参：`context`
   - 出参：核心数据结果或无返回
   - 作用：完成主数据落库
-> `XXXPostService.handleAfterPersist(context)`
   - 入参：`context`
   - 出参：无或处理结果
   - 作用：处理后置逻辑、通知或异步动作
-> 返回 `Result`
   - 作用：把最终执行结果返回给调用方
```

## 当前实现特征与限制

这一节适合总结“现在是怎么设计的”和“现在还不支持什么”。

推荐写法：

```md
## 当前实现特征与限制

### 当前实现特征

- 主流程和动作流程已经分开
- 批量场景走异步处理
- 公共查询和公共动作统一沉到 Support 层

### 当前限制

- 当前只支持同步结果返回，不支持逐条结果追踪
- 某些通知逻辑仍是占位实现
- 某些状态语义和业务口径未完全一致
```

## 边界处理与特殊说明

这一节记录最容易遗漏、最容易踩坑、最影响稳定性的边界规则。

建议写：

- 重复提交如何处理
- 状态不合法如何处理
- 是否支持批量
- 是否存在递归保护
- 是否依赖异步或外部系统
- 某些特殊兼容逻辑为什么不能删

## 维护与扩展指引

这一节写给后续维护者，说明后面改代码时应先看哪里、先守住什么、不建议先动什么。

推荐写法：

```md
## 维护与扩展指引

### 维护时建议先看

1. 入口层
2. 主服务
3. 核心上下文构建方法
4. 公共支撑层

### 维护时要先守住的不变量

- 同一条业务数据不能重复进入同一运行状态
- 主流程成功后必须完成必要的后置处理
- 批量链路不能破坏当前的事务边界

### 不建议优先改动的部分

- 批量主流程骨架
- 当前稳定运行的异步分片结构
- 已经用于兼容历史逻辑的关键保护代码
```

## 当前已知问题与优化建议

这一节用于区分：

- 当前确认存在的问题
- 可以考虑的优化方向

推荐写法：

```md
## 当前已知问题与优化建议

### 当前已知问题

- 批量任务缺少逐条结果查询入口
- 某些异常场景日志不够完整

### 优化建议

- 增加逐条结果持久化
- 增强失败补偿和日志追踪
```

## 错误处理现状

这一节写“当前错误怎么处理”，不是写理想方案。

推荐写法：

```md
## 错误处理现状

### 主流程阶段

可能失败的场景包括：

- 参数缺失
- 数据不存在
- 状态不合法
- 配置不完整

当前处理方式：

- 严格模式下整批回滚
- 非严格模式下单条失败、其他继续

### 异步阶段

可能失败的场景包括：

- 分片执行失败
- 异步任务超时
- 后置处理失败

当前处理方式：

- 当前分片回滚
- 回退逐条重试或记录失败日志
```

## 待完善项

这一节只写还没完成、后续明确要推进的内容。

推荐写法：

```md
## 待完善项

- 补充某个异常分支处理
- 补充批量场景测试覆盖
- 补充某个异步结果查询能力
```

## 已知风险或注意点

这一节只写当前已经确认的风险和注意点，不写模糊猜测。

推荐写法：

```md
## 已知风险或注意点

- 某配置缺失会导致主流程失败
- 某异步链路缺少逐条结果追踪
- 某外部接口失败后只能记录日志，不能自动补偿
```

## 总结

这一节只做收束，不要重复前文全部内容。

建议写：

- 当前系统已经形成的核心能力
- 当前最需要注意的限制
- 后续维护时最值得优先关注的点

## 使用要求

1. 只在用户明确要求维护开发文档时使用该模板。
2. 文档默认描述“当前实际实现”，不要把猜测写成事实。
3. 默认按以下顺序组织：文档标题、功能目标、当前完成情况、文档目的、代码分层与职责、核心类与职责、关键方法说明、关键对象字段状态、完整流程链、方法调用链路、当前实现特征与限制、边界处理与特殊说明、维护与扩展指引、当前已知问题与优化建议、错误处理现状、待完善项、已知风险或注意点、总结。
4. 类和方法很多时，只写核心类和核心方法。
5. 方法调用链路中的每一行都补中文解释，写清入参、出参和作用。
6. 不要把某个项目特有的业务章节写死到公共模板里。
7. 如果新增了文档中原本没有记录的功能，直接补写到对应章节。
